## 1. 底层区块链持续扫描

### 1.1 整体扫描架构

BEpusdt 的区块扫描采用 **全链路扫块 + 本地匹配** 的方式，而非对每个地址发起单独的查询请求。

```
┌─────────────────────────────────────────────────────────────────┐
│                       区块链网络节点 (RPC)                        │
│    Tron gRPC              EVM JSON-RPC (BSC / ETH / Polygon…)   │
└──────────┬──────────────────────────┬───────────────────────────┘
           │ 每 ~3s 拉取最新区块高度     │ 每 ~5s 拉取最新区块高度
           ▼                          ▼
┌──────────────────┐        ┌──────────────────────┐
│ syncBlocksForward│        │  syncBlocksForward   │
│  (Tron 正向同步)  │        │  (EVM 正向同步)        │
└────────┬─────────┘        └──────────┬───────────┘
         │  待扫区块编号入队              │  待扫区块范围入队 (批量最多10个)
         ▼                              ▼
┌────────────────────────────────────────────────────────┐
│              blockScanQueue  (无界异步队列)               │
│          chanx.UnboundedChan  · 最大堆积 100 个          │
└──────────────────────┬─────────────────────────────────┘
                       │  消费
                       ▼
┌────────────────────────────────────────────────────────┐
│              blockDispatch (goroutine worker pool)      │
│     Tron: 2 个并发 Worker     EVM: 3 个并发 Worker       │
└──────────────────────┬─────────────────────────────────┘
                       │  解析完成
                       ▼
┌────────────────────────────────────────────────────────┐
│                    transferQueue                        │
│              提取出的 transfer 记录入队                   │
└──────────────────────┬─────────────────────────────────┘
                       │
                       ▼
┌────────────────────────────────────────────────────────┐
│              orderTransferHandle (订单匹配)              │
└────────────────────────────────────────────────────────┘
```

> 源码位置：`app/task/tron.go` · `app/task/evm.go` · `app/task/transfer.go`

---

### 1.2 扫块两种方向

| 方向                              | 触发条件               | 目的                          |
|---------------------------------|--------------------|-----------------------------|
| **正向扫描** (`syncBlocksForward`)  | 定时循环               | 实时跟随链上最新区块，保证新交易不漏掉         |
| **反向扫描** (`syncBlocksBackward`) | 程序首次启动，或检测到已有待支付订单 | 补扫程序 **未运行期间** 产生的历史区块，防止漏单 |

反向扫描回溯量的计算逻辑（以 Tron 为例）：

```
回溯区块数 = (当前时间 - 最早待支付订单创建时间) / 3秒(波场出块速度) + 冗余5个区块
```

---

### 1.3 对比「单地址轮询」的优势

市面上许多同类项目的做法是：对每个收款地址，定时调用 `getTransactionsByAddress` 之类的 API 来查询该地址的交易记录。

BEpusdt 采用 **全量扫块** 的策略，具体区别如下：

```
┌─────────────────────────────────────────────────────────────────┐
│  传统「单地址轮询」模式                                            │
│                                                                  │
│  地址A ──→ getTransactions(A) ──→ API 调用 ×1                   │
│  地址B ──→ getTransactions(B) ──→ API 调用 ×1                   │
│  地址C ──→ getTransactions(C) ──→ API 调用 ×1                   │
│      ⋮                                 ⋮                        │
│  地址N ──→ getTransactions(N) ──→ API 调用 ×N                   │
│                                                                  │
│  总 API 调用次数 = 地址数量 × 轮询频率                             │
└─────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│  BEpusdt「全量扫块」模式                                          │
│                                                                  │
│  区块 #N ──→ getBlock(N) ──→ API 调用 ×1                        │
│              │                                                   │
│              ├─ 解析出所有 transfer                               │
│              └─ 本地内存匹配：地址A / 地址B / 地址C / … 地址N      │
│                                                                  │
│  总 API 调用次数 = 区块数 (与地址数量无关)                          │
└─────────────────────────────────────────────────────────────────┘
```

**核心优势：**

| 维度       | 单地址轮询            | 全量扫块 (BEpusdt)    |
|----------|------------------|-------------------|
| API 调用量  | 与地址数线性增长         | **固定**，与地址数无关     |
| RPC 节点压力 | 高（易触发限流）         | **低**，易满足免费节点限额   |
| 地址扩展性    | 增加地址 = 增加 API 压力 | **随意添加**地址，无额外成本  |
| 实时性      | 受限于轮询间隔          | 与出块速度对齐（秒级）       |
| 漏单风险     | 较高（轮询窗口期）        | **低**，区块连续无跳跃     |
| 历史回溯     | 依赖节点接口能力         | **内建支持**，自动补扫历史区块 |

---

### 1.4 Tron 与 EVM 扫块的技术差异

| 项目          | Tron              | EVM 系列                               |
|-------------|-------------------|--------------------------------------|
| 通信协议        | gRPC（长连接复用）       | HTTP JSON-RPC                        |
| 出块时间        | ~3 秒              | 1~12 秒不等                             |
| 扫块触发频率      | 每 3 秒             | 每 5 秒                                |
| 每次批量区块数     | 1 个               | 最多 10 个                              |
| Token 事件解析  | 手动解析合约 ABI 数据     | `eth_getLogs` + ERC-20 Transfer 事件签名 |
| Gas Free 支持 | ✅ 特殊解析 GasFree 合约 | —                                    |

## 2. 订单匹配原理

### 2.1 匹配键：地址 + 交易类型

系统在内存中维护一张等待匹配的订单索引表，Key 由 **收款地址 + 交易类型** 组成：

```go
// app/task/transfer.go · getAllWaitingOrders()
key := t.Address + string(t.TradeType)
// 例：
// "TRX123...abcusdt.trc20"
// "0xABC...defusdt.bep20"
```

每当从区块中解析出一笔 `transfer`，就以 `RecvAddress + TradeType` 为 Key 在索引表中查找候选订单，再进一步做金额比对。

---

### 2.2 同地址同金额冲突的处理

**创建订单时**的冲突检测与自动规避（`app/model/order.go · CalcTradeAmount`）：

```
创建订单
    │
    ▼
查询当前所有「等待中」和「确认中」的订单
    │
    ▼
构建冲突锁表：lock["地址+金额"] = true
    │
    ▼
计算本次订单金额（法币 ÷ 汇率，保留精度）
    │
    ▼
检查 lock["地址A + 金额X"] 是否存在？
    │
    ├─ 不存在 ──→ 直接分配，订单创建成功
    │
    └─ 已存在 ──→ 金额 += 1个原子精度单位，重新检查
                  最多尝试 100 次，超出则报错

```

**示例：**

```
已有订单：地址A / 10.01 USDT
新订单金额计算结果也是：10.01 USDT
    → 冲突！自动调整为 10.02 USDT
再有同地址订单：
    → 调整为 10.03 USDT
```

**扫块匹配时**的防重（`app/task/transfer.go · orderTransferHandle`）：

同一批次 `transfer` 中，一旦某笔交易匹配了订单，该订单会被**立即从内存 Map 中移除**，确保批次内其他 transfer 不会重复命中同一订单。

---

### 2.3 金额匹配三种模式

系统提供三种金额匹配策略，可在后台「支付设置」中配置（`payment_match_mode`）：

```
┌──────────────────────────────────────────────────────────────────┐
│  模式一：Classic（经典精确匹配）                                    │
│                                                                   │
│  规则：实际到账金额字符串 == 订单金额字符串（完全相等）               │
│                                                                   │
│  订单金额：10.01                                                   │
│  到账金额：10.01  ✅ 匹配                                          │
│  到账金额：10.011 ❌ 不匹配                                         │
│  到账金额：10.02  ❌ 不匹配                                         │
│                                                                   │
│  适用场景：对收款金额精确控制，防止多付混淆                           │
└──────────────────────────────────────────────────────────────────┘

┌──────────────────────────────────────────────────────────────────┐
│  模式二：HasPrefix（前缀匹配 / 允许多付）                           │
│                                                                   │
│  规则：实际到账金额以订单金额为前缀，允许多付小数                     │
│                                                                   │
│  订单金额：10.01                                                   │
│  到账金额：10.01  ✅ 匹配（完全等于）                               │
│  到账金额：10.011 ✅ 匹配（以 10.01 为前缀）                        │
│  到账金额：10.019 ✅ 匹配（以 10.01 为前缀）                        │
│  到账金额：10.02  ❌ 不匹配                                         │
│                                                                   │
│  适用场景：用户手动转账时可能多付小数，允许容错                       │
└──────────────────────────────────────────────────────────────────┘

┌──────────────────────────────────────────────────────────────────┐
│  模式三：RoundOff（数值修约 / 四舍五入）                            │
│                                                                   │
│  规则：将两边金额按精度四舍五入后比较                                │
│  精度取：订单金额的实际小数位 与 币种标准精度 中较大值                 │
│                                                                   │
│  订单金额：10.01 (精度2位)                                          │
│  到账金额：10.014 → 四舍五入 → 10.01  ✅ 匹配                      │
│  到账金额：10.015 → 四舍五入 → 10.02  ❌ 不匹配                    │
│  到账金额：10.009 → 四舍五入 → 10.01  ✅ 匹配                      │
│                                                                   │
│  适用场景：链上转账因精度截断导致微小误差，允许容错                   │
└──────────────────────────────────────────────────────────────────┘
```

> 源码位置：`app/task/transfer.go · amountMatch()`

---

## 3. 地址独占模式

### 3.1 什么是地址独占

普通订单：分配 **地址 + 金额**，系统根据「地址+金额」双重维度匹配交易。

地址独占模式：**只分配地址，不指定金额**。该地址在订单存续期间被独占，任何金额的收款都会触发回调。

**触发条件：** 创建订单时不传 `amount` 字段（或传 `0`）。

```go
// app/handler/epusdt/epusdt.go · CreateTransaction()
AddressLocked: req.Amount == 0, // 金额为0 → 独占模式

// app/model/trade.go · BuildOrder()
AddressLocked: p.Money.IsZero(), // 零值订单 → 地址锁定
```

---

### 3.2 地址独占分配逻辑

```
创建独占订单
    │
    ▼
LockTradeAddress(addressPool, tradeType)
    │
    遍历所有可用地址
    │
    ├─ 地址A：查询是否有「等待中」或「确认中」的独占订单？
    │          → 有 → 跳过（该地址已被占用）
    │          → 无 → ✅ 分配此地址，Amount = "0"
    │
    ├─ 地址B：同上检测 …
    │
    └─ 全部地址已被占用 → 返回错误「暂无可用钱包地址」
```

> 独占模式下，同一地址不会同时存在两个有效的独占订单。

---

### 3.3 独占订单的匹配行为

```go
// app/task/transfer.go · orderTransferHandle()
// 独占订单跳过金额匹配，任何入账均可命中
if !o.AddressLocked && !amountMatch(t.Amount, o.Amount, string(o.TradeType)) {
continue // 普通订单：金额不符则跳过
}
// 独占订单：o.AddressLocked = true，直接跳过此 if，进入有效期检查
```

收款后，系统用实际到账金额和订单创建时锁定的汇率，**动态计算法币金额**并写入订单：

```go
// app/model/order.go · MarkConfirming()
if o.AddressLocked {
rate, _ := decimal.NewFromString(o.Rate)
o.Amount = amount.String() // 实际到账加密货币数量
o.Money = rate.Mul(amount).String() // 折算法币金额
}
```

---

### 3.4 独占模式 vs 普通模式对比

```
┌──────────────────────────────────────────────────────────────────┐
│  普通订单（带金额）                                                 │
│                                                                   │
│  创建 → 分配：地址A + 金额 10.01 USDT                             │
│  收款条件：必须向地址A 转入 10.01 USDT（±匹配模式容差）              │
│  回调内容：包含约定金额                                             │
└──────────────────────────────────────────────────────────────────┘

┌──────────────────────────────────────────────────────────────────┐
│  独占订单（不带金额）                                               │
│                                                                   │
│  创建 → 分配：地址A（独占，金额=0）                                  │
│  收款条件：向地址A 转入 任意金额（大于监控最小金额即可）              │
│  回调内容：包含实际到账金额 + 汇率折算法币金额                        │
│  典型用途：充值场景、不定额收款、钱包监控                             │
└──────────────────────────────────────────────────────────────────┘
```

## 4. 汇率浮动说明

### 4.1 汇率来源

BEpusdt 通过 **CoinGecko API** 定期同步加密货币对多种法币（CNY / USD / JPY / EUR / GBP）的实时汇率。

```
定时任务（默认间隔 3600 秒）
    │
    ▼
GET https://api.coingecko.com/api/v3/simple/price
    ?ids=tether,usd-coin,tron,binancecoin,ethereum
    &vs_currencies=cny,usd,jpy,eur,gbp
    │
    ▼
写入数据库 bep_rate 表
（含 raw_rate 原始汇率 + syntax 浮动语法 + 最终 rate）
```

> 同步间隔可在后台「系统设置」中调整，最短可设置为数秒级。

---

### 4.2 汇率浮动语法

除直接使用 CoinGecko 的原始汇率外，系统支持为每种 **加密货币 × 法币** 对单独设置浮动语法，实现灵活调价。

浮动语法规则（配置键：`rate_float_<CRYPTO>_<FIAT>`，如 `rate_float_USDT_CNY`）：

```
┌────────────┬───────────────────────────────────────────────────────┐
│  语法格式   │  说明                                                  │
├────────────┼───────────────────────────────────────────────────────┤
│  7.25      │  固定汇率，直接使用该数值，忽略 CoinGecko 原始价格        │
├────────────┼───────────────────────────────────────────────────────┤
│  ~1.05     │  乘以系数：最终汇率 = 原始汇率 × 1.05（上浮 5%）         │
├────────────┼───────────────────────────────────────────────────────┤
│  +0.1      │  加固定值：最终汇率 = 原始汇率 + 0.1                     │
├────────────┼───────────────────────────────────────────────────────┤
│  -0.05     │  减固定值：最终汇率 = 原始汇率 - 0.05                    │
├────────────┼───────────────────────────────────────────────────────┤
│  （空）     │  不浮动，直接使用 CoinGecko 原始汇率                     │
└────────────┴───────────────────────────────────────────────────────┘
```

**示例计算：**

```
CoinGecko 当前 USDT/CNY = 7.2456

语法 ~1.03  → 最终汇率 = 7.2456 × 1.03 = 7.4630
语法 +0.20  → 最终汇率 = 7.2456 + 0.20 = 7.4456
语法 7.30   → 最终汇率 = 7.30（固定，不参考市价）
语法 （空）  → 最终汇率 = 7.2456（原始市价）
```

> 最终汇率保留 4 位小数。源码：`app/model/rate.go · ParseFloatRate()`

---

### 4.3 汇率在订单生命周期中的作用

```
创建订单
    │
    ├─ 从 bep_rate 获取最新汇率（含浮动计算）
    │
    ├─ 法币金额 ÷ 汇率 = 加密货币金额（按精度取整）
    │
    └─ 汇率快照写入订单 rate 字段（后续不再变更）
            │
            ▼
    订单支付期间
            │
            汇率波动不影响本次订单
            │
            ▼
    收款成功
            │
            ├─ 普通订单：以创建时约定金额为准，回调包含法币金额
            │
            └─ 独占订单：以实际到账金额 × 快照汇率计算法币金额
```

> 汇率快照机制保证了**下单瞬间确定交易金额**，不因支付期间行情波动而产生金额争议。

---

### 4.4 创建订单时指定汇率（高级用法）

API 支持通过 `rate` 参数**强制指定本次订单汇率**，此时将忽略数据库中的汇率记录：

```json
POST /api/v1/epusdt/create
{
  "order_id": "xxx",
  "amount": 100.0,
  "fiat": "CNY",
  "rate": "7.30",
  ...
}
```

若传入 `rate` 参数，则浮动语法也不再生效，直接以传入值为准。

## 附录：关键配置项速查

| 配置键                     | 默认值       | 说明                                            |
|-------------------------|-----------|-----------------------------------------------|
| `payment_match_mode`    | `classic` | 金额匹配模式：`classic` / `has_prefix` / `round_off` |
| `payment_timeout`       | `1200`    | 订单默认超时时间（秒），范围 180~3600                       |
| `rate_sync_interval`    | `3600`    | 汇率同步间隔（秒）                                     |
